# Icon Loader 项目需求文档

## 1. 项目目标

本项目用于构建一个 Web 页面，展示基于 icon 生成的 Loader 动画。

系统需要把本地 icon 资产转换为通用资源，再基于这些资源生成等待动画。Gemini 流式回复链路保留，作为真实等待场景和后续基于 thinking 过程驱动动画的扩展入口。

## 2. 使用场景

用户在 Web 页面中输入问题，点击提交后进入等待状态。

等待期间页面同时发生两件事：

1. Gemini API 持续返回流式回复内容。
2. 页面展示 Icon Loader 动画。

当流式回复结束后，页面展示完整回复，并结束或弱化 Loader 动画。

## 3. 核心需求

### 3.1 页面输入

- 页面提供一个文本输入框。
- 页面提供一个提交按钮。
- 用户点击提交按钮后，使用输入内容调用 Gemini 真实 API。
- 请求进行中时，需要避免重复提交导致多个请求状态互相覆盖。
- 输入为空时，不应发起请求。

### 3.2 流式回复

- 系统需要接入 Gemini 真实 API。
- 回复内容需要以流式方式逐步展示。
- 流式回复开始后，页面进入 loading 状态。
- 流式回复结束后，页面退出 loading 状态。
- API 调用失败时，需要展示明确的错误状态，并停止 Loader 动画。

### 3.3 Icon Loader

Icon Loader 必须满足以下要求：

- 基于本地 icon 生成的通用资源播放。
- 动态变化。
- 每次请求看到的内容不完全一样。
- 当前只保留原像素组装逻辑。
- 不展示真实、有用、可被误解为推理过程的信息。
- 不伪装成真实进度、真实工具调用、真实链路追踪或真实模型思考。

当前动画行为：

1. icon 资源由构建脚本转换为 `64 * 64` 彩色像素资源。
2. Loader 运行时将资源转换为 `24 * 24` 展示点阵。
3. 运行时从资源池中不放回抽取一轮资源。
4. 每轮内每个 icon 只播放一次。
5. 每个 icon 按 seed 稳定选择填充顺序。
6. 一轮结束后派生新轮次 seed，并重新洗牌。

### 3.4 未来扩展边界

后续可以基于 Gemini thinking 过程结果影响动画表现，但必须遵守：

- thinking 数据必须在边界层清洗后进入核心逻辑。
- 动画可以表达抽象状态变化，不得假装展示真实进度。
- Domain 层保持纯粹，不直接依赖 Gemini 协议或服务端返回结构。
- 新动画模式必须基于同一份 Icon Loader 通用资源，而不是绕过资源契约另起一套。

## 4. 非目标

本项目当前不做以下内容：

- 不做移动端适配。
- 不使用 Mock 大模型回复作为最终实现。
- 不展示真实推理过程。
- 不展示真实进度百分比。
- 不展示真实工具调用状态。
- 不保留旧的非 Icon Loader动画。
- 不构建完整聊天系统。
- 不做用户登录、历史记录、权限管理等后台能力。

## 5. 用户流程

1. 用户打开 Web 页面。
2. 用户在输入框中输入问题。
3. 用户点击提交按钮。
4. 前端校验输入非空。
5. 前端调用 Gemini 真实 API。
6. 页面进入 loading 状态，并展示 Icon Loader。
7. 页面逐步展示 Gemini 返回的流式回复。
8. 流式回复完成后，页面退出 loading 状态。
9. 如果请求失败，页面展示错误信息并结束 loading 状态。

## 6. 状态设计

页面至少需要显式建模以下状态：

- `idle`：初始状态，未提交请求。
- `loading`：请求进行中，正在接收流式回复。
- `success`：请求完成，已有完整回复。
- `error`：请求失败。

状态不应通过散落的字符串随意判断，应使用强类型集中约束。

## 7. 错误处理

需要覆盖以下错误场景：

- 输入为空。
- Gemini API 请求失败。
- Gemini API 返回异常。
- 流式连接中断。
- 用户在请求未完成时重复提交。
- Icon Loader manifest 缺失或格式非法。
- Icon Loader 资源文件加载失败。

错误发生时：

- 页面退出 loading 状态。
- Loader 动画停止或切换为静态等待态。
- 页面展示可理解的错误信息。
- 不应吞掉错误，也不应只打印日志后继续假装成功。
- Loader 资产错误不应影响 Gemini 回复展示。

## 8. 技术约束

- 项目类型：Web 网页。
- 运行目标：桌面 Web 浏览器。
- 前端技术栈：React、TypeScript、Vite。
- 动画运行时：PixiJS。
- 大模型接口：Gemini 真实 API。
- 回复模式：流式回复。
- 移动端：不做适配。

## 9. 可维护性要求

实现需要注意：

- API 调用逻辑与动画逻辑分离。
- 页面状态显式建模。
- Icon Loader 类型、资源、事件和渲染器具备清晰边界。
- Gemini 返回的外部数据需要在进入核心逻辑前做边界校验。
- 业务逻辑和测试工具需要分文件存放。
- 相关文档需要随实现同步更新。

## 10. 验收标准

项目完成后应满足：

- 用户可以在 Web 页面输入问题并提交。
- 系统调用 Gemini 真实 API。
- Gemini 回复以流式方式逐步展示。
- 回复 loading 过程中展示 Icon Loader 动画。
- Icon Loader 动画按 `24 * 24` 点阵展示。
- 每次请求的 Loader 表现存在可见差异。
- Loader 不展示真实推理信息或虚假进度。
- 请求失败时页面有明确错误状态。
- 桌面浏览器下页面展示正常。
